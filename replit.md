# ProCV — Your Personal Career Consultant

## Overview

ProCV is a full-featured PWA designed to help users build, manage, and download professional CVs with advanced AI assistance. The project aims to be a personal career consultant, leveraging AI for CV generation, cover letter creation, ATS analysis, and interview preparation. It integrates market intelligence to create highly relevant and impactful CVs, providing a comprehensive toolkit for job seekers. The business vision is to empower individuals in their job search journey, offering a competitive edge through AI-driven personalization and optimization.

## System Architecture

ProCV is built as a React 19, TypeScript, and Vite 6 PWA, styled using Tailwind CSS and custom CSS variables.

### UI/UX Decisions
- **Brand Name**: ProCV — "Your Personal Career Consultant"
- **Color Scheme**: Primary Deep Corporate Navy (`#1B2B4B`), Accent Muted Gold (`#C9A84C`), Background Warm Off-White (`#F8F7F4`).
- **Typography**: Playfair Display (headings) and DM Sans (body).
- **Templates**: Offers 32 customizable CV templates across various categories (professional, modern, creative, academic, technical) with color customization options.
- **WYSIWYG PDF**: Utilizes `html2canvas` for precise PDF generation matching on-screen previews.

### Technical Implementations
- **AI CV Generation**: A two-phase process:
    1. **Market Research**: Google Gemini 2.0 Flash (via `marketResearch.ts`) with Google Search grounding researches live market trends (top skills, ATS keywords, expected tools). This can adapt to scenarios with no JD, short hints, or full JDs.
    2. **CV Generation**: Groq's `llama-3.3-70b-versatile` generates the CV using the gathered market intelligence. Supports honest, boosted, or aggressive generation modes.
- **Multi-model CV pipeline (Apr 2026)**: The 8 generic `cv-engine-worker` LLM calls in `geminiService.ts` (CV generation, validator, humanizer audit, plain-text humanizer, profile parser, cover letter, plus the two CV-gen pre-size/fallback paths) now route through the worker's **tiered endpoint** with task-specific models instead of a single paid 70b. Task → model mapping lives in `TIERED_MODEL_MAP` (`cv-engine-worker/src/index.ts`): `cvGenerate` → Llama 4 Scout 17B (paid, ~3x cheaper output than 70b), `cvGenerateLong` → GLM 4.7 Flash 131K context (free), `cvAudit`/`cvValidate` → Llama 4 Scout, `parser` → Mistral Small 3.1 24B (free), `humanize`/`coverLetter` → Hermes-2 Pro 7B (free). The main CV generation path uses a **race endpoint** (`POST /api/cv/race-llm`, helper `workerRaceLLM` in `services/cvEngineClient.ts`) that fires `cvGenerate` and `cvGenerateLong` in parallel server-side and returns whichever finishes first. Cost guardrail: Workers AI cannot cancel in-flight calls, so racers all complete and any paid model in a race is billed regardless of who wins — only race when at least one candidate is free. Worker code changes require `cd cv-engine-worker && npm run deploy` (uses `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID` env vars).
- **Groq removed from default path (Apr 27 2026)**: `groqChat()` in `services/groqService.ts` is now redirected to the cv-engine-worker's tiered endpoint **first**. Mapping: `GROQ_LARGE` (Llama 3.3 70b versatile) → `cvGenerate` task (Llama 4 Scout 17B), `GROQ_FAST` (Llama 3.1 8b) → `general` task (Llama 3.1 8B free). Every existing Groq call site in the codebase (15+ in `geminiService.ts` and elsewhere) now lands on Cloudflare Workers AI with no code change at the call sites. Groq + Cerebras keys remain only as **last-resort offline fallbacks** — they fire only if the worker is unreachable AND the user has a key configured. Default user flow never touches Groq, no API key required.
- **PDF page-break + WYSIWYG cleanup across core templates (Apr 28 2026)**: Two related downloaded-PDF bugs reported. (1) Big blank space at the bottom of page 1 with everything after Summary jumping to page 2 — caused by the blanket rule `section { page-break-inside: avoid }` in `services/getCVHtml.ts`, which forced any too-tall section (Experience) to be pushed entirely to the next page. Replaced with finer-grained rules: `h2,h3,h4 { break-after: avoid }` keeps headings glued to following content, `li { break-inside: avoid }` keeps bullets atomic, and a new opt-in `[data-pdf-keep] { break-inside: avoid; page-break-inside: avoid; }` lets templates mark individual entries (one job, one degree) to stay together while letting the surrounding section flow. (2) Multiple template previews did not match their downloaded PDFs — root cause was responsive Tailwind classes that diverged between the small editor preview viewport and the 794×1123 PDF viewport. Cleanup applied to **5 core templates** (`TemplateProfessional.tsx`, `TemplateClassic.tsx`, `TemplateModern.tsx`, `TemplateExecutive.tsx`, `TemplateCorporate.tsx`): replaced responsive padding (`p-8 sm:p-12` → `p-12`, `p-10 sm:p-14` → `p-14`), removed `hidden sm:inline` separators in contact lines (Professional), removed `grid-cols-1 sm:grid-cols-2` (Classic references → always 2 columns), and added `data-pdf-keep="true"` to every per-entry wrapper (header block, each experience entry, each education entry, each project entry, each publication entry, each reference entry where present). The remaining ~30 templates still use their original responsive layouts — they can be migrated on demand if users report similar mismatches.
- **PDF generation slowness fixed (Apr 28 2026)**: Investigated reports of ~2-min PDF downloads. Two root causes found and fixed: (1) `services/getCVHtml.ts` was fetching every Google Font WOFF2 file (often 20–40 per request) with **no timeout** and **no caching** — a single slow `fonts.gstatic.com` response could stall `Promise.all` indefinitely, and every download click re-fetched the same files. Added a 4s timeout on the CSS fetch, 3s timeout per font file (via `AbortSignal.timeout`), and a module-level `_fontCssMemo` `Map<href, Promise<css>>` so subsequent clicks are instant. Also exported `clearFontEmbedCache()` for theme/template changes. (2) `server-pdf.cjs` used `waitUntil: 'networkidle'` which always adds ≥500ms of network-silence wait — wasteful when fonts are pre-embedded as base64 data-URIs. Switched to `waitUntil: 'load'`, wrapped `document.fonts.ready` in a `Promise.race` with a 2.5s ceiling, and dropped the post-render stabilisation timer from 600ms → 100ms. PDF renders should now be a few seconds instead of minutes.
- **Worker DB usage clarified (Apr 28 2026)**: Investigated user concern that `cv-engine-worker` was feeding example/seed CV rows from D1 into LLM prompts. Confirmed it does NOT — the D1 + KV stores are a **style dictionary only**: `cv_verbs` (strong verbs), `cv_openers`, `cv_sentence_structures`, `cv_rhythm_patterns` (e.g. `["short","medium","long"]`), `cv_voice_profiles`, `cv_field_profiles`, `cv_banned_phrases`, `cv_request_telemetry`. These are sent to the LLM as constraints ("use verbs from this list", "follow this rhythm pattern") via `buildBriefData()` (lines 448–594 of `cv-engine-worker/src/index.ts`), never as example user content. Real CV data comes from the frontend's `profile`/`jd` payload. The worker is explicitly stateless re: user data (lines 1015–1016 of `index.ts`). No code changes needed.
- **OpenRouter + Together.ai added as free fallbacks (Apr 28 2026)**: Cloudflare Workers AI's 10,000 daily Neuron quota is **account-wide** (proven via probing 5 different free models — all returned the same 4006 error). To survive this, two more free upstream providers were added with their own separate daily quotas: **OpenRouter** (free tier — Llama 3.3 70B, Qwen 2.5 72B, Gemma 3 27B) and **Together.ai** (free tier — Llama 3.3 70B Turbo Free). Both are OpenAI-compatible and reuse the existing `openAiCompatChat()` helper via a generic `callOpenAiCompatChain()` runner that auto-cycles models on 404/402. Final fallback chain in `groqChat()` is now: **Worker CF AI → Groq → Cerebras → OpenRouter → Together.ai → Claude → Gemini**. New keys: `openrouterApiKey`, `togetherApiKey` on `ApiSettings` (encrypted by `KeyVault.SENSITIVE_FIELDS`), runtime cache slots `openrouter`/`together` in `RuntimeKeys.ts`, settings UI cards in `SettingsModal.tsx` (orange & pink branded), and `testProviderConnection()` extended to the two new providers.
- **Section-parallel CV endpoint (Apr 27 2026)**: New worker route `POST /api/cv/parallel-sections` (helper `workerParallelSections` in `services/cvEngineClient.ts`) lets the frontend split CV generation into N section-specific prompts (summary, skills, education, experience, projects, …) that fan out concurrently inside a single Worker request. Each section picks its own task/model from `TIERED_MODEL_MAP` (free 8B for simple sections, paid Scout 17B for bullet-heavy ones) and auto-retries the `cvFallback` task (Mistral Small 3.1 24B free) on per-section failure. Results come back keyed by section name with per-section model + ms + fellBack diagnostics.
- **`generateCV()` now uses section-parallel as PRIMARY (Apr 27 2026)**: The main CV generation path in `services/geminiService.ts` builds the existing megaprompt as a shared preamble (with `${CV_DATA_SCHEMA}` stripped out) and fans it out across 4–5 section calls (`cvSummary`, `cvSkills`, `cvExperience`, `cvEducation`, optional `cvProjects`). Each section gets its own narrower schema instruction in plain English. Results are merged into a complete `CVData` object that drops into the existing polish pipeline unchanged. **Wall-clock**: 5–10s typical (vs 19–30s for the legacy single-megaprompt path). Fallback chain: section-parallel → race endpoint → groqChat (which itself routes through the worker tiered endpoint). The legacy single-megaprompt race path is preserved as backup but rarely fires.
- **Scout 17B silent-empty bug (Apr 27 2026)**: Llama 4 Scout 17B on Workers AI silently returns empty responses when the user prompt contains literal JSON-example blobs like `{"experience":[{"jobTitle":"..."}]}` — the model triggers some safety/parsing path and outputs nothing. Workaround applied to all section-parallel instructions in `geminiService.ts`: schemas are now described in plain English (e.g. "an array where each item is an object with these string fields: company, jobTitle, dates, …") instead of JSON examples. Verified Scout reliability went from 0% → ~80% (with auto-fallback to Mistral 24B catching the rest). The legacy `CV_DATA_SCHEMA` constant still uses literal JSON examples — it's only consumed by the legacy fallback race path which uses GLM 4.7 Flash (immune to this bug).
- **OpenRouter free-model chain refresh (Apr 28 2026)**: The Cloudflare Worker hits a hard 10,000-neuron-per-day cap on the free CF plan; once exhausted the worker returns HTTP 502 `4006: you have used up your daily free allocation of 10,000 neurons` and the daily reset is at 00:00 UTC. When that happens the app falls through to the OpenRouter free-tier path (`callOpenAiCompatChain` in `services/groqService.ts`). The previous chains in `OPENROUTER_LARGE_CHAIN` / `OPENROUTER_FAST_CHAIN` referenced two retired model IDs (`qwen/qwen-2.5-72b-instruct:free`, `mistralai/mistral-small-3.1-24b-instruct:free`) which 404'd and broke the chain. Refreshed both chains against the live `https://openrouter.ai/api/v1/models` catalog. New LARGE chain: `nvidia/nemotron-3-super-120b-a12b:free` → `qwen/qwen3-next-80b-a3b-instruct:free` → `meta-llama/llama-3.3-70b-instruct:free` → `openai/gpt-oss-120b:free` → `nousresearch/hermes-3-llama-3.1-405b:free` → `google/gemma-3-27b-it:free`. New FAST chain: `nvidia/nemotron-nano-9b-v2:free` → `openai/gpt-oss-20b:free` → `google/gemma-3-12b-it:free` → `meta-llama/llama-3.2-3b-instruct:free`. The chain runner already auto-cycles on 404/402 errors, so any future deprecation gracefully steps to the next ID. SettingsModal description updated to list the new lineup. **OpenRouter only fires when the CF Worker is offline AND the user has set an OpenRouter key** — under normal operation the app never touches it.
- **AI CV Toolkit**:
    - **CV Checker**: Provides ATS score, keyword match analysis (semantic matching via Cloudflare Workers AI), strengths/weaknesses, and a "Fix & Regenerate" loop.
    - **Humanization Audit**: Uses `cv-engine-worker` with Cloudflare Workers AI (`@cf/meta/llama-3.3-70b-instruct-fp8-fast`) to enforce voice consistency and identify banned phrases, falling back to Groq.
    - **Quality Polish (single source of truth)**: One `runQualityPolishPasses` helper in `geminiService.ts` is THE place where post-Groq CV polish lives. Every generation path — `generateCV`, `improveCV` (Auto Optimize), and `optimizeCVForJob` — calls into it, so all three flows produce CVs at strict parity. The helper runs (in order): humanizer (with corrupt-metric revert) → bullet-count enforcer → banned-phrase filter → optional profile customSections/sectionOrder carry-through → experience sort → deterministic purify (with telemetry hook for caller-owned reporting) → optional voice-consistency enforcement against the CV-engine brief (with revert) → **second deterministic purify pass** (catches anything the voice-fix LLM reintroduced — orphan metrics, weird verbs, etc.) → finalize (source-fidelity vs profile or vs source CV) → pronoun safety net. Bullet-count strategy is selectable: `profile-pointcount` (Generate path — honours user's `pointCount` from `UserProfile`) or `preserve-cv` (Improve / Optimize paths — never silently changes structure). Every AI step is wrapped so a worker / Groq hiccup never aborts the polish; the deterministic passes still run and the user gets a finished CV. Tune CV quality here — nowhere else.
    - **Bullet hygiene helpers** (in `services/cvPurificationPipeline.ts`): `stripOrphanMetrics` catches placeholder bullets the LLM forgot to fill (e.g. `KES ,000`, `a %`, `the % retention rate`, currency-followed-by-orphan-thousands). `rewriteWeirdOpeners` swaps low-quality CF Workers AI verbs (`Re-framed`, `Re-positioned`, `Moderated`, `Advocated for`, `Discussed`, `Engaged`, `Liaised`, `Utilised`, `Leveraged`, `Spearheaded`, `Orchestrated`) for canonical pool verbs. Both are wired into `polishBullet` so every bullet that flows through purify gets them.
- **Orphan-metric leak fix (Apr 28 2026)**: Three regression cases were leaking through to the final summary and experience bullets on both Worker AI and the fallback chain. (1) `stripOrphanMetrics` ran BEFORE `formatNumbers` in `polishBullet` / `polishSummary`, so legitimate `"increased by 5 %"` (digit-space-%) had its `%` stripped because the lookbehind only checks one char back (the space). Fixed by reordering: `formatNumbers` now runs first to collapse `"5 %"` → `"5%"` and `"$ 1,000"` → `"$1,000"`, so the orphan sweep sees only true orphans. (2) `%word` (no whitespace, letter immediately after — e.g. `"%growth in revenue"`) was missed because the old final sweep had `(?!\w)` which gave up. (3) `.%` (period before %, no digit — e.g. `"delivered .% improvement"`) was missed because `(?<![\d.])` blocked any period. Replaced the old final sweep with `out.replace(/(?<!\d)[.,;:]\s*(?=%)/g, '')` then `out.replace(/(?<!\d)\s*%\s*/g, ' ')` — first kills any stray punctuation immediately before an orphan `%` (skipped when part of a real number like `0.5%`), then strips the orphan `%` and surrounding whitespace regardless of what follows. Also extended the dangling-preposition cleanup to anchor at field start (`(?:\s+|^)`) and added `through|via|while|during|across|within|including|featuring` so `"reduced costs through %"` → `"reduced costs"` and `"by approximately %"` → `""` instead of leaving `"by approximately"` or `"by"`. Verified 23/23 cases (regression + legitimate metric preservation: `0.5%`, `15% YoY`, `$50,000`, `Mr. Smith`).
    - **Resilience: session-level circuit breakers**: The frontend assumes any infra dependency may be wedged. `services/groqCacheClient.ts` opens a single circuit on the first 5xx/timeout/network error and short-circuits all subsequent `lookupGroqCache`/`storeGroqCache` calls for the rest of the session. `services/cvEngineClient.ts` keeps a per-endpoint `deadEndpoints` Set covering `getJSON`, `postJSON`, `workerLLM`, and `workerTieredLLM`, so one dead worker route never costs the user repeated round-trips. Each circuit logs `Circuit opened for <endpoint> (<reason>) — skipping subsequent calls this session.` exactly once. **Pre-size routing**: in `geminiService.ts` main generation, prompts where `system + user > 90,000` chars skip the Groq attempt entirely (it would 413) and route straight to `workerLLM`, saving ~3s per generation.
    - **Vision Extract**: `cv-engine-worker` (`@cf/meta/llama-3.2-11b-vision-instruct`) is used for extracting text from images, falling back to Gemini for PDFs and other vision tasks.
    - **Word Import**: `mammoth` parses `.docx` files, and Groq extracts structured profile data.
- **PDF Generation**: All CV downloads route through a single shared service (`services/cvDownloadService.ts`) that renders the live preview DOM via headless Chrome — Playwright (local dev, port 3001) → Cloudflare `resume-pdf-worker` (production). What you see in the preview is what you get in the PDF, for all 32 templates. The legacy hand-coded jsPDF templates in `services/pdfService.ts` are no longer used for CV rendering and are queued for cleanup; cover-letter rendering and the PDF merger still use `pdfService.ts` and have not been migrated yet.
- **Storage**: A write-through triple-layer storage system utilizing `localStorage`, `IndexedDB`, and optional Google Drive sync.
- **Multi-Profile Management**: Users can create, switch, rename, color-code, and delete multiple CV profiles.
- **PWA Capabilities**: Offline capable and installable through a service worker.
- **ATS Keyword Embedding**: Critical JD tier-1 keywords are embedded as selectable-but-invisible text in generated PDFs to improve ATS parsing.
- **Self-improving vocabulary**: The `cv-engine-worker` includes features like "Leak Queue" for identifying new banned phrases, an AI Auditor for novel AI-isms, and a Voice Tester for fine-tuning voice profiles.

### System Design Choices
- **Client-Side Application**: The main application logic resides entirely on the client-side.
- **CV Engine Worker**: A Cloudflare Worker (`cv-engine-worker`) handles the deterministic aspects of CV generation, such as voice profiles, verb pools, banned phrases, and field detection. It uses Cloudflare Workers AI, D1, and KV for its operations. This worker also performs post-generation enforcement and AI auditing.
- **Authentication**: Google OAuth (PKCE flow) for Google Drive sync and other Google services. Microsoft OAuth for OneDrive integration.
- **API Key Management**: API keys are stored in the browser's `localStorage` and managed via a Settings modal.
